Product System + Deploy

它不是一个页面，而是一套产品系统。

架构页负责回答三个问题：为什么这套产品能落地、AI 在系统里放在哪些位置、官网为什么要独立成 `site/` 并通过 GitHub Actions 部署。

三层结构，分别承担不同任务

把官网、桌面端和业务系统拆开，既能保持用户体验，也方便继续扩展与部署。

体验层

`site/` 负责官网叙事与产品展示，`client/` 负责 Tauri 客户端体验。一个面向获客，一个面向日常运营。

client/
官网 / 桌面端

业务层

`app/` 提供 FastAPI 路由、业务模块和发布链路。CLI、Tauri、MCP 都调用同一套能力，不会形成多套逻辑。

FastAPI
modules/
publish worker

数据与执行层

`data/app.db`、`assets/`、`crawler/` 和浏览器登录态共同支撑图库、自动发布和趋势复盘，不是一个只会生成文字的孤立前端。

AI 在系统中的位置

架构上最重要的一件事，是让 AI 作为系统能力进入不同模块，而不是散落在几个按钮里。

Profile Analyzer

AI 分析账号人设

帮助构建长期创作上下文，减少后续每轮手工补背景。

Library Analyzer

AI 理解图片素材

把图片转成标签、卖点和可写角度，进入图库与 Prompt 构建器。

Prompt Builder

AI 组合上下文

把人设、素材和榜样账号重新组织成可执行的草稿输入。

Editor Shortcuts

AI 参与编辑流程

在正文编辑器里承担改写、压缩和补标签动作，而不是脱离编辑器独立存在。

Analytics Insight

AI 提炼复盘结论

让发布结果不只停在趋势图，而是重新回流到下一轮内容策略。

MCP Tools

AI 进入 Agent 协作

通过 MCP，外部 Agent 可以直接调用业务动作，而不是只能看静态页面。

为什么官网源码放在 `site/`

它本质上是产品官网，而不是文档。目录组织应服务维护性，部署策略再单独解决。

源码结构

site/
  index.html
  product.html
  architecture.html
  shared.css
  main.js
  assets/
  screens/

client/
app/
docs/

GitHub Pages 部署

.github/workflows/site-pages.yml

on:
  push:
    paths:
      - site/**

deploy:
  upload-pages-artifact:
    path: site

这样目录职责和部署方式分离，官网作为独立前端目录维护，发布时再交给 Actions 处理。

为什么这比单点 AI 工具更有产品感

因为它不是把 AI 套在一个页面外面，而是通过 GUI、CLI、FastAPI、发布链路和数据回流形成闭环。

真实执行链

从草稿到发布状态查询都是系统动作，不只停在“生成一段文案”。

真实数据回流

复盘结果会重新影响看板建议、选题和写法，而不是独立孤岛。

真实协作接口

MCP 让外部 Agent 也能直接调用业务能力，而不只是人工操作 GUI。

官网、产品和系统层都已经拆开了。

接下来继续迭代时，就不需要再在一个超长单文件里同时维护营销叙事、交互逻辑和模拟应用结构。
